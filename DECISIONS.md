# DECISIONS.md — Smart Cuts Architecture Choices

One-line entries. Add new entries at top, dated.

## 2026-05-28 (Session 4 — Bright Cuts + Calculator + Drum picker + units + more)
- **Bright Cuts (display only) — repo stays smart-cuts** — Michael's wife preferred the original "Bright Cuts" branding. Renamed display name everywhere (header, manifest, README, copy-plan text) but **kept the GitHub repo slug + Pages URL as `smart-cuts/`** so existing phone PWA installs don't break and bookmarks survive. The CLAUDE.md note about Bright Cuts being the historical name is now flipped back — Smart Cuts was the rename that's been undone.
- **Drum picker as the universal measurement input** — every length/width/thickness input opens an iOS-style bottom-sheet drum wheel on tap. Two scroll-snap columns: whole inches + fraction (Imperial 1/16th) or cm + mm (Metric). Inputs are marked `readonly + inputmode=none` so the OS keyboard never opens — drum is the ONLY input mechanism. Saves typing typos and forces 1/16" / 1mm snap. Tradeoff: desktop users can't type fast; resolved by 1) drum scrolls with mouse-wheel too, 2) numbers can be very fast to dial in. Worth the consistency.
- **1/16" snap (not 1/8")** — user request, after building the drum at 1/8". Finer control matters for woodworking. The frac display uses mixed glyphs: ⅛ ¼ ⅜ ½ ⅝ ¾ ⅞ for common, ¹⁄₁₆ ³⁄₁₆ ⁵⁄₁₆ etc. for odd sixteenths.
- **Imperial / Metric — full bidirectional, storage normalized to inches** — internal state is always inches; display unit only affects what shows in inputs and `dim()` output. Toggle snapshots current inches, switches unit, re-renders. Save/load converts to/from inches on the way through localStorage so unit toggles never corrupt data. fromInches() snaps to 1/16" so mm→in roundtrips don't accumulate float drift.
- **Calculator: lowest-denominator power-of-2 fractions up to 64ths** — not just 1/16" snap. So 2 1/16" − 3/32" returns the EXACT 1 31/32" (32nds), not snapped down to 1 30/32 = 1 15/16". A `_formatInchFrac()` helper finds the best fraction with denominators in [2,4,8,16,32,64] then reduces to lowest terms. Engineering-grade math output, not woodworking-approximation.
- **Calculator math model: chained operations** — operator key auto-applies the pending `a op b` if both operands set, treating result as the new `a`. So `2 + 3 + 4 = 9` chains naturally; `(10 − 2) × 3 = 24` continues from the result. `calcJustComputed` flag distinguishes "fresh value clears chain" vs "operator continues chain" — typing a new value after `=` starts fresh, but tapping an operator after `=` continues with the result.
- **Header gets its own 🧮 calculator shortcut** — the calculator is reached often enough that it deserves a permanent shortcut, not just a tab. Sits next to the ⋯ menu on both phone and desktop. Copper accent border so it's discoverable.
- **Lumber Prices as a dedicated tab, not buried in Cutting Board** — the inline price input on the species picker is a *quick adjustment* per session; the Lumber Prices tab is the *persistent source of truth*. `priceFor(speciesName)` looks at the saved overrides first, then the SPECIES_PALETTE default. Active session species pick up changes live via `updateSpeciesPriceFromTable()` so the cost math updates instantly when you edit a price from either screen.
- **Pattern picker = bottom-sheet with thumbnails (not buttons)** — 7 buttons were already cramped on phone, and we want to add more patterns. Replaced with a single "current pattern" preview card → opens the same drum-overlay shell with a 2-col grid of pattern cards (each a labeled mini SVG). Adding a new pattern is now: 1 PATTERN_CATALOG entry + 1 `_patternThumb()` recipe + the actual cutting-render logic.
- **Implicit-link stripe widths** — when two strips share the same width, editing one ripples to all matched siblings. Implicit because Michael said "if I change one, change the matched ones" — no explicit linking UI needed. Hold Alt to bypass the ripple. Tradeoff: can't have two intentionally-separate strips at the same width; rare enough not to matter.
- **Ratio templates use formula `x · ¼ · ½ · ¼ · x`** — Michael's example. Implemented as `faceLayoutPreset('sym5')` etc.: compute base `x` from the panel width and the ratio sum (e.g., for sym5: `x = W / 3` because `2x + 2(x/4) + x/2 = 3x`). Last strip absorbs any drift from 1/16" snap so totals match exactly.
- **Tree silhouettes as 7 reusable SVG shapes, not per-species art** — drawing 56 unique tree shapes is content overkill; 7 stylized shapes (broad / oval / narrow / conifer / cypress / spreading / birch-with-bark) cover the major silhouette families. Each species in WOOD_TREE_DATA references one. Birch is the exception because the chalk-white peeling bark is iconic.
- **WOOD_TREE_DATA as a separate lookup, not extended on WOOD_ENCYCLOPEDIA** — keeps the existing encyclopedia untouched (which is critical data — hundreds of lines hand-curated). New fields live in their own object; the wood card consults both. Easy to roll back, easy to incrementally fill in remaining species.
- **iOS status bar: `status-bar-style="default"` (was black-translucent)** — black-translucent caused white-on-cream contrast bugs in light mode (status bar text overlapped the parchment header). `default` reserves a separate strip — iOS shows status bar in its own color matching theme-color meta. Users with the OLD PWA install need to re-add to home screen to pick up the meta change. New meta theme-color tags handle light/dark.
- **Plan title XSS / quote bug — escapeHtml() everywhere a user string lands in an HTML attribute** — plan titles include `"` from dimension marks (`Cabinet 30"×24"d`). When piped raw into `value="..."` or `alt="..."` the first `"` closes the attribute. Wrapping every title insertion (in attribute *and* body contexts) with `escapeHtml()` prevents both display corruption and any actual XSS via user-typed titles.
- **Tab strip auto-scrolls active tab into view** — with 10 tabs the strip overflows on phone. switchTab() now centers the active tab in the scroll container after the .active class swap. `scrollLeft` direct assignment instead of `scrollTo({behavior: smooth})` because smooth animation on flex-wrap container was buggy.
- **Home page Design section uses cols-3 grid on phone** — same problem as the cutting-board presets (3 tiles in 2-col grid = 2+1 wrap with orphan). Fix: dedicated `.home-tile-grid.cols-3` modifier for the Design section so 3 tiles stay in a single row on phone. Then Calculator joined Design as a 4th tile, returning Design to 2×2.
- **Light-mode input background DARKER than page bg** — counterintuitive but works for parchment aesthetic. Was `#fcfaf3` (almost white on cream), now `#e6d6b6` (deeper tan with `#b69d72` border). Inputs sit IN the page, not floating ABOVE it. Focus state lifts back to bg color with accent border.

## 2026-05-27 (Session 3 — Sparky overhaul + Plans)
- **Wood-forward palette swap** — moved from charcoal + cyber-yellow to warm walnut/espresso (dark) + parchment/oak (light), keeping the same CSS variable names so everything else inherits. Dark mode `--bg` went from `#141414` to `#18120c`, accent from `#e8a838` to honey-amber `#d4a056`. Replaced raw `rgba(232,168,56,...)` everywhere with the new amber RGB; same for the harsh blue `rgba(74,158,255,...)` → muted slate. Rationale: this is a woodworking app — the chrome should feel like the material, not a generic SaaS dashboard.
- **Mobile-first home + app-launcher tiles** — flipped the home grid from desktop-default `repeat(auto-fill, minmax(280px, 1fr))` (which dropped to 1 column on phone) to **2 cols on phone, 3 on tablet, 4 on desktop**. Each tile is a square Sparky-icon card with title-only on phone, full description+CTA only at desktop breakpoint (≥980px). Felt like a launcher, not a marketing page.
- **Header collapses to ⋯ menu on phone** — the four utility buttons (Sign in / Theme / Print / Reset) lived inline before, which forced a second row on phone. Now they're wrapped in `.header-tools` that's inline-flex on desktop and a dropdown popover (anchored under a 38px ⋯ button) on `<820px`. Brand subtitle hidden on phone — it's already on the home hero.
- **Square Sparky General app icon, no name/text** — the source "Sparky - General" PNG (1587×1437, wide rectangle) was center-padded to a square with the cream background, then resized to 180/192/512/32. The bare character with hammer + thumbs-up is the brand mark on home screen + favicon + app hero. Plain (no caption) per user.
- **19 Sparky character icons in `sparky/` folder** — all sourced from the user's `Downloads/Sparky - *.png` set, padded to square + resized to 256px PNG (~70-110KB each, ~1.7MB total — lazy-loaded so only on-screen tiles fetch). Decided to ship all 19 even though we only use ~10 currently; the rest are reserved for future placements (Plans / supplies / finishing / safety).
- **Plans = its own tab + data model, not part of Projects** — a Plan is a *design* (what you intend to build). A Project is a *build record* (what you actually made, with photos). Storing them as separate collections (`woodshop-plans-v1` vs `woodshop-projects-v1`) means the catalog naturally grows in two directions and the user can see "designed but not yet built" plans as a backlog. Plans carry an optional `projectId` to link them to the project after the build is done. Could have shoved Plans into the Project record as a child field, but separating them keeps the design step and the build step decoupled — you can design a plan, never build it, and it's still a useful catalog entry.
- **Save Plan = explicit button, not autosave** — added a `+ Save Plan` button to the finished-preview block on Cutting Board and Furniture rather than auto-snapshotting on every change. Rationale: the user is iterating with live preview; auto-save would create dozens of garbage plans. Explicit save = the design moment the user *committed* to.
- **Plan title is auto-generated** — `${species} ${pattern} ${L}×${W}` for boards, `${type} ${W}×${H}×${D}d` for furniture. User can Rename via the plan card. Avoided forcing a title prompt on save — keep the friction low.
- **Assigning a plan can also create a new project** — the modal has a "+ New project" path that pre-fills the project form (title, type, dims, species) from the plan and auto-links after save via a temporary `_pendingPlanId` flag on the draft. Saves the user from re-typing all the spec when the plan IS the project.
- **Plans tab inserted between Optimizer and Projects in the nav** — sits with the action tools (Cutting Board / Furniture / Optimizer → Plans → Projects), so the flow reads left-to-right: design → save → build → log.
- **Toast component reused across the app** — a simple absolute-positioned `.toast` div anchored above the safe-area inset. Created on demand by `showToast(text, onTap)`, auto-hides after 3.2s, optionally clickable to navigate. Used for "Plan saved" and "Plan assigned to project". Avoided a heavier notification framework.

## 2026-05-27
- **Light mode is the default on mobile, dark on desktop** — user request. `theme` init checks `matchMedia('(max-width: 820px)')` only when there's no saved `woodshop-theme`; a manual toggle still wins and persists. Rationale: phone usage (outdoors/shop, bright screens) reads better light; desktop stays dark.
- **Mobile polish for app-like use** — sticky header on ≤820px (tabs always reachable), iOS safe-area insets via `env(safe-area-inset-*)` on header + body (notch/status-bar/home-indicator), bigger tap targets (tabs + header actions), `touch-action: manipulation` + transparent tap highlight, and hid the "Live preview" label on phones to save vertical space.
- **Cloud sync = Supabase (reverses the no-backend decision)** — user needs cross-device project sync, so localStorage-only is no longer sufficient. Plan: single `smartcuts_state` table (one JSONB row per user per collection: projects / supplies / shop_photos), RLS `auth.uid() = user_id`, email magic-link auth, supabase-js from CDN. Push-on-save + pull-on-login + Realtime + visibility-pull; offline edits flagged dirty and flushed before pulls so they're never clobbered. App still works fully offline via localStorage cache. Hosted in a **separate free Supabase org** Michael owns (kept out of Validator Digital infra; a 6th VD-org project would've cost $10/mo). Code is committed but inert until `SUPABASE_URL`/`SUPABASE_ANON_KEY` constants are filled.
- **Renamed "Bright Cuts" → "Smart Cuts"** — user decision. Renamed display name, GitHub repo slug, and Pages URL (`michael-vdam.github.io/smart-cuts/`). Local folder stays `~/code/woodshop/`. GitHub auto-redirects the old `bright-cuts` URL, but the new URL is canonical — re-add to home screen from it.
- **PWA / Add to Home Screen** — added `manifest.webmanifest` (standalone display, portrait, theme `#141414`) + `apple-touch-icon` + apple-mobile-web-app meta tags. Makes the app installable full-screen on phone with its own icon. Relative paths (`./`, `icon-*.png`) so it works under the `/smart-cuts/` Pages subpath.
- **Single PNG logo, retired the 4-variant SVG picker** — user wanted the Gemini "Michael Makes It Work" lightbulb as the real brand mark everywhere (header, home hero, home-screen icon). Removed `LOGO_VARIANTS`/`applyLogo`/`setLogo`/`renderLogoPicker` + the "Choose a logo" UI + `woodshop-logo` localStorage key. Simplicity over customization.
- **Icons via `sips`, not inline** — iOS Add-to-Home-Screen needs a real image file for apple-touch-icon (data URIs are unreliable). Source logo is 2048×2100 → center-cropped to square (`sips -c`) → resized to 180/192/512/32. This is the first time the app isn't strictly single-file; icons + manifest are a deliberate, small exception for installability.
- **Logo rendered with `border-radius:50%`** — the source PNG is a cream circle on a dark square; circular crop hides the square corners cleanly on both light and dark themes without re-cutting the image.

## 2026-05-26
- **Single HTML file, no build step** — easier hosting (works from `file://` or any static host), offline-capable, no dependencies to break. Tradeoff: 4700 lines is getting big, but search/navigation via grep is fine.
- **Vanilla JS, no framework** — same reason: no build, no node_modules, no version churn. Code is readable to anyone who knows JS.
- **localStorage for ALL user data** — no backend means no server costs, no auth complexity, no privacy concerns. Tradeoff: per-device data, no cross-device sync. Acceptable for personal app; Export/Import or cloud sync can layer on later.
- **Photos auto-resized to 1200px max JPEG 0.82** — keeps localStorage under quota (~5-10MB). User can fit 20-50 projects with photos before hitting cap. Quality is sufficient for portfolio reference.
- **Brand color `#e8a838` (warm gold-yellow)** — matches the lightbulb in the user's "Michael Makes It Work" logo. Light mode uses slightly darker `#c47a1a` for contrast on white.
- **Dark mode default** — most woodworkers/makers prefer dark UIs. Light mode optional, persists in localStorage.
- **App originally named "Bright Cuts"** — tied to lightbulb logo (bright/ideas) + woodworking (cuts). Renamed to "Smart Cuts" on 2026-05-27 (see top of file).
- **Removed Plan Build / Generate Parts buttons** — live preview made them redundant. "Simplicity wins" — user feedback.
- **Stock Lumber section hidden in cutting board UI** — defaults to 4/4 × 6" × 36" lumber. Hiding reduces cognitive load. Inputs still in DOM for math + save/load. Custom users who need different stock would have to know to override via JS (acceptable for power users).
- **Standard mode hides L/W inputs** — preset picks them. Only Thickness editable. Simplest possible flow for the 90% case.
- **Custom mode hides Standard Sizes** — clean separation of modes. Power user only sees what they need.
- **Strip Pattern: removed Quick Layouts dropdown, kept # of Strips** — user feedback "doesn't add value, only distracts." # of strips + per-strip editor is enough.
- **Grain lines run HORIZONTAL in face-grain preview** — matches real wood grain direction (along the strip's length). Originally drew vertical (bug, fixed).
- **Wood Library uses Mimms Lumber inventory (56 species)** — user's actual local source in Nashville. Categorized: domestic / figured / exotic.
- **Dynamic Project finish/stain dropdowns** — hardcoded common options + user's My Supplies in oil-wax/stain/finish categories. User can add Walrus Oil to supplies, it shows up everywhere relevant.
- **Quiz best-score persistence per category** — keyed by category, only updates when beating previous best. Shows "★ Best X%" on welcome.
- **Logo as inline SVG, not external image** — kept single-file architecture intact; 4 variants user could switch between. SUPERSEDED 2026-05-27: replaced by a single external PNG logo (see top of file) for real-brand consistency + home-screen icon.
- **Mobile breakpoints at 820px and 480px** — covers tablet + phone. Workspace stacks input → output vertically. Tabs scroll horizontally.
- **GitHub Pages hosting (not Netlify)** — user has GitHub account, prefers git push workflow over drag-and-drop. Version control for free. Auto-deploys ~30 sec.
- **Repo public** — no secrets in code. Easier for Pages, simpler permissions.
- **Author email = noreply GitHub email** — doesn't leak personal email in commit history of a public repo.
- **3-file Claude convention** (CLAUDE.md / PROGRESS.md / DECISIONS.md) — copied from Core's vd-session-lifecycle pattern. Future sessions read these at start to orient quickly.

## Reverted / things we tried that didn't work
- **3D Cube hexagonal SVG rendering** — attempted full hex projection. Math was right but visual was busy. Reverted to flat tiles + accurate build notes (60° miter cuts). Future work to render real hexagons.
- **Auto-fetching Amazon/HD product images** — researched, can't legally embed (copyright + CORS + URL rot). Switched to manual photo upload with category emoji fallback.
- **stockBoardsNeeded vs sum-of-per-species buffered** — initial math had rounding drift (ceil applied at both levels). Now total = sum of per-species buffered, consistent.
- **Wood library 'Oak' / 'Zebrawood' encyclopedia entries** — old keys preserved when palette renamed to 'White Oak' / 'Zebra Wood'. Dead data, not removed (no functional impact).
