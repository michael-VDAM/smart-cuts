## 2026-05-28 (Session 5 — Photo annotation on Projects)
- **Non-destructive annotation model: normalized 0–1 coords stored alongside the photo, rendered as an SVG overlay** — the photo dataURL stays untouched; annotations live in `project.annotations = [{kind, color, ...}]`. Rendering uses a `<svg viewBox="0 0 100 100" preserveAspectRatio="none">` absolutely-positioned over the `<img>`. Coords multiply by 100 to enter viewBox space. Tradeoff: circles render as ellipses on non-square photos because of the stretch. Acceptable for MVP — arrows + text carry most of the annotation load, and a true circle on a 4:3 photo would mean storing aspect ratio per project or using a measuring-based rendering pass. Skipped both for simplicity.
- **Two-click annotation flow (no drag, no canvas)** — Arrow: tap start, tap end. Circle: tap center, tap edge. Text: tap, then OS prompt. Drag-to-draw would have meant pointer-event listeners + touch handling + a render-during-drag preview pipeline. Two-click is finger-friendly, works identically on touch and mouse, and the "tap an existing mark to delete it" mode rides on the same primitive. Pending state stored as a small marker dot at the first click so the user can see it.
- **SVG markers for arrowheads, one per color, declared once in a `<defs>`** — `<marker id="annot-arrow-red" ...>` × 4 colors. Each `<line marker-end="url(#annot-arrow-red)">` references its color's marker. Cheaper than computing arrowhead polygons in JS and re-orienting them; the browser handles `orient="auto-start-reverse"` for free. Minor visual quirk on non-square photos (the marker orients in viewBox space, not displayed space) — acceptable.
- **Delete tool reuses the same SVG with per-item pointer-events** — when tool === 'delete', each annotation gets `pointer-events: auto; cursor: pointer; onclick=delete`. Other tools render items with `pointer-events: none` so clicks pass through to the canvas for new placements. No mode-switching of event listeners, no overlay layer juggling.
- **Editor is a modal, viewer is inline; both share `annotSvgMarkup()`** — the editor modal needs the canvas + toolbar UI, but the detail view just needs the same overlay rendered read-only over the photo. Both call the same helper with `deletable: false` (detail view) or `deletable: true` (delete-tool mode). One source of truth for how annotations look.
- **Annotate button only appears when the project has a photo** — guarding the button at render time avoids the "you opened the annotator on a project with no photo and got an empty modal" failure mode. The button check `${p.photo ? ... : ''}` is a single conditional.
- **Cancel discards, Done commits — editor state is a clone, not a reference** — `annotState.items = (p.annotations || []).map(a => ({...a}))` so edits during a session don't mutate the project record until Done. This is the standard non-destructive editor pattern and is the difference between a tool you trust and a tool you don't.

## 2026-05-28 (Session 5 — Patterns: Chevron + Basketweave + Diagonal)
- **Three new end-grain patterns share the existing grid-render architecture** — `tileColorFor(col, row, p)` does all the color logic; `svgFinishedBoard` adds per-pattern grain overlays. Adding a pattern means: PATTERN_CATALOG entry + `_patternThumb()` recipe + `tileColorFor` branch + optional grain-line block in svgFinishedBoard + Phase 2 step-description entry + patternLabel map. ~30 lines per pattern. No new infra needed because Wave 5 already generalized the picker.
- **Chevron color formula: `((col + floor(row/2)) % n)`** — every row-pair holds a stripe-shifted version of the previous, so adjacent species form continuous diagonal runs that mirror at row-pair boundaries → perfect Vs. The mirrored grain lines (`/` for even pair, `\` for odd) carry the chevron silhouette even when species contrast is low. Different from herringbone (`(floor(col/2) + row) % n`) which interlocks the legs at the half-cell level instead of stacking row-pairs.
- **Basketweave uses 2×2 mega-blocks, not 2-species checker** — `idx = (floor(col/2) + floor(row/2)) % n`. The visual contrast is grain orientation per block: horizontal blocks render with 3 evenly-spaced horizontal lines (side-grain look), vertical blocks render with the standard end-grain concentric dots. Works in a single species (orientation alone carries it) or two contrasting ones (extra punch). Skipped a 4-cell-direction approach because every-other-block is cleaner and easier to glue up in real life.
- **Diagonal uses 2-cell-wide anti-diagonal bands: `floor((col+row)/2) % n`** — geometrically accurate to a 45° crosscut on a stripe panel (band width ≈ stripe width × √2). 3-cell-wide bands looked more "stripey" at preview zoom but mis-represent the build process. Chose accuracy over visual punch — the thumbnail makes the pattern intent obvious before the user picks it.
- **Build-step descriptions emphasize how the pattern differs from standard end grain** — each new pattern has a single short paragraph explaining the cut sequence (45° miters for chevron, square-block rotation for basket, 45° crosscut for diag). Avoided generic "arrange the slabs" copy because these patterns each have a non-obvious construction step that determines whether the user can actually build them in their shop.

## 2026-05-28 (Session 5 — Parametric stripe + input bug + haptic removed)
- **Parametric stripe formulas are the strip-width model now** — every strip carries a `formula` string ("x", "x/2", "2x+y", "x*phi", "1.5") and the `width` field is a *derived* 1/16"-snapped value. Variables live in `stripVars` (defaults to `{x: 1.5}`) and the built-in constant `phi = 1.618033988749`. Editing a variable propagates through every strip that references it; same formula = implicit link (replaces the old same-width-linking rule). Tradeoff: parametric presets (Halves, Sym5, Sym7, Golden, Golden Accent) can drift up to 1/16" off `finalW` because we refuse to lie about the formula. The total bar shows the drift honestly + offers an Auto-fit button. Non-parametric presets (Solid, Single, Paired, Triple, Etsy, Uniform) still snap-fit the last strip to absorb 1/16" drift because there's no formula to honor. Auto-fit always converts strips back to literal-number formulas (locks in the layout).
- **`evalFormula()` is a regex-gated `new Function` evaluator** — character whitelist `[0-9a-z.+\-*/()\s]` rejects anything else upfront, then implicit-multiplication inserts (`1.5x → 1.5*x`, `2(x+1) → 2*(x+1)`, `(x+y)(z) → (x+y)*(z)`), then unknown identifier check, then `new Function` runs the expression. Invalid formulas return NaN → strip width 0 + `.invalid` CSS class on the input. Safer than `eval()` because the whitelist makes injection impossible; faster than a hand-rolled parser. Rejected a Pratt parser as overkill for woodworking math.
- **Old saves migrate by backfilling formula = String(width)** — `ensureStripFormulas()` runs after load and converts any pre-formula strip into a literal-number-formula strip. `recomputeAllStripWidths()` then re-evaluates everything from formula → width. Result: identical numbers, but now every strip is editable as a formula going forward. Zero data loss.
- **Input clobber bug fix: `stripCount` uses `onchange`, not `oninput`** — old handler `oninput="setStripCount(parseInt(this.value)||1)"` clobbered the field the moment it went blank (setStripCount→refreshStripList→countEl.value = stripPattern.length), so deleting "4" to type "3" gave you "13". Switched to `onchange` (commits only on blur/Enter) + an `onblur` fallback that restores the last value if user leaves it blank + the writeback in refreshStripList now skips when `document.activeElement === countEl`. Belt-and-suspenders, but the bug was annoying enough to deserve all three. Other number inputs (pf-coats/hours/salePrice, species price) don't have the bug because nothing else re-writes their `.value` between keystrokes — they're left alone.
- **Haptic removed entirely** — Michael not feeling it on iPhone PWA. Apple has refused to ship the Web Vibration API for ~10 years; `navigator.vibrate()` is a silent no-op on iOS Safari. The misleading toggle + 7 vibration call sites got cut. Android users lose a working feature, but the user is iOS-only — accepting that tradeoff. Easy to restore (one commit) if iOS support ever arrives or Android users matter.

## 2026-05-28 (Session 4 Wave 5 — Custom species + Timer + Print + Haptic)
- **Custom species via `allSpecies()` indirection, not by mutating SPECIES_PALETTE** — SPECIES_PALETTE is a const that other code (defaults, encyclopedia lookups, init) reads at startup. Mutating it would risk side effects. Instead added `customSpecies = []` as a separate array and `allSpecies()` returns the concat. Where it matters (species picker, lumber prices list, priceFor lookup) we call `allSpecies()`. Where it doesn't (default species pair, initial board setup) we keep SPECIES_PALETTE. Same effect, lower blast radius.
- **Build timer model: `{accumulated, startedAt}` not a single live counter** — separates "time committed across pauses" from "in-progress segment." Total = accumulated + (now - startedAt if running). Survives reloads cleanly because accumulated is a frozen ms value at pause time; on resume we set startedAt to current time and let the math add up. A naive "elapsed += interval" approach drifts during long-running sessions if the interval fires unevenly (page sleeps, tab backgrounded) — this delta-from-start approach is monotonic and accurate.
- **Only one plan timer runs at a time, with auto-pause on new start** — woodworkers don't multi-task across plans. Trying to track concurrent timers invites confusion. Starting a new timer commits the running one's elapsed to its accumulated and starts fresh. Clear, predictable.
- **Timer hours auto-roll into Project on assign, only if Project's hours field is empty** — preserves any manually-entered hours. If you've already typed "12" in the project form, we don't overwrite. If the field is blank, we fill it. Best of both: data flows when convenient, doesn't fight intentional input.
- **Print stylesheet hides EVERY button selector wholesale (`button { display: none !important; }`)** — than enumerate each one (Save Plan, Copy, Send, etc.). The print is read-only by definition. Side effect: all .btn-styled elements in any tab's content disappear in print. Acceptable — the print is for the current tab, and that tab's content already has no interactive elements that should print.
- **Print header is a single global element on body, updated by `beforeprint`** — rather than injecting per-tab print headers. Centralized, gets the active tab's name from `document.querySelector('.tab.active').dataset.tab` lookup right before print. One place to maintain.
- **Haptic feedback: `navigator.vibrate()` only, no audio** — visceral, immediate, doesn't make noise in a quiet workshop. Audio adds asset weight + permission prompts + annoyance risk. Toggle in ⋯ menu defaults to ON; users who don't want it (or are on iOS where it's a no-op) can disable. Sound deferred per Michael ("i don't like voice stuff for now" — and audio cues bordered the same concern).
- **Haptic ms values are intentionally small (3-20)** — 3ms for drum snap (each scroll-snap event), 5ms for tab/operator/pattern (light tick), 15ms for calc equals (satisfying click), 20ms for plan save (confirmation pulse). Anything over 50ms feels like an error vibration. The pattern is "you can feel it but barely register it" — kinaesthetic confirmation, not announcement.

## 2026-05-28 (Session 4 — Bright Cuts + Calculator + Drum picker + units + more)
- **Bright Cuts (display only) — repo stays smart-cuts** — Michael's wife preferred the original "Bright Cuts" branding. Renamed display name everywhere (header, manifest, README, copy-plan text) but **kept the GitHub repo slug + Pages URL as `smart-cuts/`** so existing phone PWA installs don't break and bookmarks survive. The CLAUDE.md note about Bright Cuts being the historical name is now flipped back — Smart Cuts was the rename that's been undone.
- **Drum picker as the universal measurement input** — every length/width/thickness input opens an iOS-style bottom-sheet drum wheel on tap. Two scroll-snap columns: whole inches + fraction (Imperial 1/16th) or cm + mm (Metric). Inputs are marked `readonly + inputmode=none` so the OS keyboard never opens — drum is the ONLY input mechanism. Saves typing typos and forces 1/16" / 1mm snap. Tradeoff: desktop users can't type fast; resolved by 1) drum scrolls with mouse-wheel too, 2) numbers can be very fast to dial in. Worth the consistency.
- **1/16" snap (not 1/8")** — user request, after building the drum at 1/8". Finer control matters for woodworking. The frac display uses mixed glyphs: ⅛ ¼ ⅜ ½ ⅝ ¾ ⅞ for common, ¹⁄₁₆ ³⁄₁₆ ⁵⁄₁₆ etc. for odd sixteenths.
- **Imperial / Metric — full bidirectional, storage normalized to inches** — internal state is always inches; display unit only affects what shows in inputs and `dim()` output. Toggle snapshots current inches, switches unit, re-renders. Save/load converts to/from inches on the way through localStorage so unit toggles never corrupt data. fromInches() snaps to 1/16" so mm→in roundtrips don't accumulate float drift.
- **Calculator: lowest-denominator power-of-2 fractions up to 64ths** — not just 1/16" snap. So 2 1/16" − 3/32" returns the EXACT 1 31/32" (32nds), not snapped down to 1 30/32 = 1 15/16". A `_formatInchFrac()` helper finds the best fraction with denominators in [2,4,8,16,32,64] then reduces to lowest terms. Engineering-grade math output, not woodworking-approximation.
- **Calculator math model: chained operations** — operator key auto-applies the pending `a op b` if both operands set, treating result as the new `a`. So `2 + 3 + 4 = 9` chains naturally; `(10 − 2) × 3 = 24` continues from the result. `calcJustComputed` flag distinguishes "fresh value clears chain" vs "operator continues chain" — typing a new value after `=` starts fresh, but tapping an operator after `=` continues with the result.
- **Header gets its own 🧮 calculator shortcut** — the calculator is reached often enough that it deserves a permanent shortcut, not just a tab. Sits next to the ⋯ menu on both phone and desktop. Copper accent border so it's discoverable.
- **Lumber Prices as a dedicated tab, not buried in Cutting Board** — the inline price input on the species picker is a *quick adjustment* per session; the Lumber Prices tab is the *persistent source of truth*. `priceFor(speciesName)` looks at the saved overrides first, then the SPECIES_PALETTE default. Active session species pick up changes live via `updateSpeciesPriceFromTable()` so the cost math updates instantly when you edit a price from either screen.
- **Pattern picker = bottom-sheet with thumbnails (not buttons)** — 7 buttons were already cramped on phone, and we want to add more patterns. Replaced with a single "current pattern" preview card → opens the same drum-overlay shell with a 2-col grid of pattern cards (each a labeled mini SVG). Adding a new pattern is now: 1 PATTERN_CATALOG entry + 1 `_patternThumb()` recipe + the actual cutting-render logic.
- **Implicit-link stripe widths** — when two strips share the same width, editing one ripples to all matched siblings. Implicit because Michael said "if I change one, change the matched ones" — no explicit linking UI needed. Hold Alt to bypass the ripple. Tradeoff: can't have two intentionally-separate strips at the same width; rare enough not to matter.
- **Ratio templates use formula `x · ¼ · ½ · ¼ · x`** — Michael's example. Implemented as `faceLayoutPreset('sym5')` etc.: compute base `x` from the panel width and the ratio sum (e.g., for sym5: `x = W / 3` because `2x + 2(x/4) + x/2 = 3x`). Last strip absorbs any drift from 1/16" snap so totals match exactly.
- **Tree silhouettes as 7 reusable SVG shapes, not per-species art** — drawing 56 unique tree shapes is content overkill; 7 stylized shapes (broad / oval / narrow / conifer / cypress / spreading / birch-with-bark) cover the major silhouette families. Each species in WOOD_TREE_DATA references one. Birch is the exception because the chalk-white peeling bark is iconic.
- **WOOD_TREE_DATA as a separate lookup, not extended on WOOD_ENCYCLOPEDIA** — keeps the existing encyclopedia untouched (which is critical data — hundreds of lines hand-curated). New fields live in their own object; the wood card consults both. Easy to roll back, easy to incrementally fill in remaining species.
- **iOS status bar: `status-bar-style="default"` (was black-translucent)** — black-translucent caused white-on-cream contrast bugs in light mode (status bar text overlapped the parchment header). `default` reserves a separate strip — iOS shows status bar in its own color matching theme-color meta. Users with the OLD PWA install need to re-add to home screen to pick up the meta change. New meta theme-color tags handle light/dark.
- **Plan title XSS / quote bug — escapeHtml() everywhere a user string lands in an HTML attribute** — plan titles include `"` from dimension marks (`Cabinet 30"×24"d`). When piped raw into `value="..."` or `alt="..."` the first `"` closes the attribute. Wrapping every title insertion (in attribute *and* body contexts) with `escapeHtml()` prevents both display corruption and any actual XSS via user-typed titles.
- **Tab strip auto-scrolls active tab into view** — with 10 tabs the strip overflows on phone. switchTab() now centers the active tab in the scroll container after the .active class swap. `scrollLeft` direct assignment instead of `scrollTo({behavior: smooth})` because smooth animation on flex-wrap container was buggy.
- **Home page Design section uses cols-3 grid on phone** — same problem as the cutting-board presets (3 tiles in 2-col grid = 2+1 wrap with orphan). Fix: dedicated `.home-tile-grid.cols-3` modifier for the Design section so 3 tiles stay in a single row on phone. Then Calculator joined Design as a 4th tile, returning Design to 2×2.
- **Light-mode input background DARKER than page bg** — counterintuitive but works for parchment aesthetic. Was `#fcfaf3` (almost white on cream), now `#e6d6b6` (deeper tan with `#b69d72` border). Inputs sit IN the page, not floating ABOVE it. Focus state lifts back to bg color with accent border.

## 2026-05-27 (Session 3 — Sparky overhaul + Plans)
- **Wood-forward palette swap** — moved from charcoal + cyber-yellow to warm walnut/espresso (dark) + parchment/oak (light), keeping the same CSS variable names so everything else inherits. Dark mode `--bg` went from `#141414` to `#18120c`, accent from `#e8a838` to honey-amber `#d4a056`. Replaced raw `rgba(232,168,56,...)` everywhere with the new amber RGB; same for the harsh blue `rgba(74,158,255,...)` → muted slate. Rationale: this is a woodworking app — the chrome should feel like the material, not a generic SaaS dashboard.
- **Mobile-first home + app-launcher tiles** — flipped the home grid from desktop-default `repeat(auto-fill, minmax(280px, 1fr))` (which dropped to 1 column on phone) to **2 cols on phone, 3 on tablet, 4 on desktop**. Each tile is a square Sparky-icon card with title-only on phone, full description+CTA only at desktop breakpoint (≥980px). Felt like a launcher, not a marketing page.
- **Header collapses to ⋯ menu on phone** — the four utility buttons (Sign in / Theme / Print / Reset) lived inline before, which forced a second row on phone. Now they're wrapped in `.header-tools` that's inline-flex on desktop and a dropdown popover (anchored under a 38px ⋯ button) on `<820px`. Brand subtitle hidden on phone — it's already on the home hero.
- **Square Sparky General app icon, no name/text** — the source "Sparky - General" PNG (1587×1437, wide rectangle) was center-padded to a square with the cream background, then resized to 180/192/512/32. The bare character with hammer + thumbs-up is the brand mark on home screen + favicon + app hero. Plain (no caption) per user.
- **19 Sparky character icons in `sparky/` folder** — all sourced from the user's `Downloads/Sparky - *.png` set, padded to square + resized to 256px PNG (~70-110KB each, ~1.7MB total — lazy-loaded so only on-screen tiles fetch). Decided to ship all 19 even though we only use ~10 currently; the rest are reserved for future placements (Plans / supplies / finishing / safety).
- **Plans = its own tab + data model, not part of Projects** — a Plan is a *design* (what you intend to build). A Project is a *build record* (what you actually made, with photos). Storing them as separate collections (`woodshop-plans-v1` vs `woodshop-projects-v1`) means the catalog naturally grows in two directions and the user can see "designed but not yet built" plans as a backlog. Plans carry an optional `projectId` to link them to the project after the build is done. Could have shoved Plans into the Project record as a child field, but separating them keeps the design step and the build step decoupled — you can design a plan, never build it, and it's still a useful catalog entry.
- **Save Plan = explicit button, not autosave** — added a `+ Save Plan` button to the finished-preview block on Cutting Board and Furniture rather than auto-snapshotting on every change. Rationale: the user is iterating with live preview; auto-save would create dozens of garbage plans. Explicit save = the design moment the user *committed* to.
- **Plan title is auto-generated** — `${species} ${pattern} ${L}×${W}` for boards, `${type} ${W}×${H}×${D}d` for furniture. User can Rename via the plan card. Avoided forcing a title prompt on save — keep the friction low.
- **Assigning a plan can also create a new project** — the modal has a "+ New project" path that pre-fills the project form (title, type, dims, species) from the plan and auto-links after save via a temporary `_pendingPlanId` flag on the draft. Saves the user from re-typing all the spec when the plan IS the project.
- **Plans tab inserted between Optimizer and Projects in the nav** — sits with the action tools (Cutting Board / Furniture / Optimizer → Plans → Projects), so the flow reads left-to-right: design → save → build → log.
- **Toast component reused across the app** — a simple absolute-positioned `.toast` div anchored above the safe-area inset. Created on demand by `showToast(text, onTap)`, auto-hides after 3.2s, optionally clickable to navigate. Used for "Plan saved" and "Plan assigned to project". Avoided a heavier notification framework.

## Reverted / things we tried that didn't work
- **3D Cube hexagonal SVG rendering** — attempted full hex projection. Math was right but visual was busy. Reverted to flat tiles + accurate build notes (60° miter cuts). Future work to render real hexagons.
